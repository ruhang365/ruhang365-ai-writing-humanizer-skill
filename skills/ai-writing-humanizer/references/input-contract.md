# Input Contract

Use this before drafting or rewriting when the prompt is underspecified.

## Five Inputs

1. **Format**: short post, long article, tutorial, review, opinion, report, proposal, script, email, or social caption.
2. **Author intent**: explain, persuade, review, criticize, document, sell carefully, or establish a judgment.
3. **Target reader**: what the reader already knows, what they misunderstand, and where they are stuck.
4. **Tone boundary**: calm judgment, field notes, personal complaint, slight irony, practical instruction, or restrained formal voice.
5. **Material source**: real experience, observed pattern, data, customer feedback, transcript, notes, speculation, or secondhand summary.

## Missing Inputs

- If the draft is usable but inputs are incomplete, proceed with explicit assumptions.
- If material source is missing, do not invent details. Mark placeholders the author can fill.
- If the target reader is "everyone", narrow it to the most likely reader from context.
- If the user asks for strong voice but gives no sample, infer lightly from the draft and ask for samples only when fidelity matters.

## Preflight Questions

Ask only the questions that materially change the output:

- Where will this be published?
- Who must feel "this was written for me"?
- Which claims come from real experience or data?
- Should the final sound like a calm editor, a founder, an operator, a reviewer, or the user's own voice?
